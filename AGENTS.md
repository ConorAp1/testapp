# AGENTS.md — TestApp

This file provides guidance for AI coding agents (e.g., OpenAI Codex, GPT-4 via Cursor, GitHub Copilot Workspace) working autonomously on the TestApp codebase. Read this fully before executing any task.

## What Is TestApp?

**TestApp** is a lightweight task management tool built for small engineering teams (3–10 people) at early-stage startups. The core problem it solves: teams waste hours per week in status meetings because there is no shared, up-to-date view of who is working on what.

**MVP scope (do not exceed this):**
1. Task creation — form to add a new task with title, description, and assignee
2. Task assignment — link a task to a team member
3. Status tracking — mark tasks as `TODO`, `IN_PROGRESS`, or `DONE`
4. Dashboard view — single page showing all tasks grouped or listed with their status and assignee

**Stack:** Next.js 14 (App Router), TypeScript (strict), Tailwind CSS, PostgreSQL, Prisma ORM

## Agent Operating Rules

These rules apply to ALL agents at ALL times. Do not deviate.

### 1. Scope Enforcement
- Work **only** on tasks directly related to the four MVP features listed above.
- If a task seems to require a feature outside MVP scope (authentication, notifications, file uploads, team management, etc.), **stop and report** instead of implementing it.
- Do not install additional npm packages unless the task explicitly requires it and no existing utility can solve the problem.

### 2. Safety — Never Do These
- Never delete or modify files in `prisma/migrations/` manually. Migrations are generated by Prisma CLI only.
- Never commit `.env.local` or any file containing real credentials.
- Never use `prisma.executeRaw` or `prisma.queryRaw` unless there is no typed Prisma alternative.
- Never use TypeScript `any`. Use `unknown` with type guards or properly typed Prisma-generated types.
- Never add `"use client"` to a component unless it genuinely requires browser APIs or React hooks.
- Never bypass the Prisma singleton — always import from `src/lib/prisma.ts`.

### 3. Before Writing Code
- Read the relevant files in context before editing. Do not assume structure.
- Check `src/lib/types.ts` for existing type definitions before creating new ones.
- Check `prisma/schema.prisma` before writing any database query to confirm field names and relations.
- If the task requires a schema change, output the updated `schema.prisma` AND the migration command to run. Do not silently change the schema.

### 4. After Writing Code
- Ensure all new TypeScript files pass strict type checking (no implicit `any`, no missing return types on exported functions).
- Ensure all Tailwind classes used exist in the default Tailwind config (no arbitrary values without justification).
- Ensure all new API routes follow the standard response shape: `{ data: T }` on success, `{ error: string }` on failure.
- If you modified `prisma/schema.prisma`, remind the user to run: `npx prisma migrate dev --name <your-migration-name>`

## Task Execution Workflow

```
1. UNDERSTAND
   - Read the task description fully.
   - Identify which MVP feature it belongs to (creation / assignment / status / dashboard).
   - Identify which files need to be read, created, or modified.

2. READ FIRST
   - Read relevant existing files before writing anything.
   - Check prisma/schema.prisma for model shapes.
   - Check src/lib/types.ts for existing types.

3. PLAN (for complex tasks)
   - List the files you will create or modify.
   - List any new types or schema changes required.
   - Flag any ambiguities before proceeding.

4. IMPLEMENT
   - Follow all coding conventions in this file.
   - Write clean, minimal code — no unused imports, no dead code.
   - Add JSDoc comments on exported functions and components.

5. VERIFY
   - Re-read your changes for TypeScript correctness.
   - Confirm all imports resolve to real files.
   - Confirm Tailwind classes are valid.
   - Confirm API routes return correct status codes and shapes.

6. REPORT
   - List every file created or modified.
   - List any commands the developer needs to run (e.g., prisma migrate).
   - Flag any assumptions made or follow-up work needed.
```

## Coding Conventions for Agents

### TypeScript
```typescript
// ✅ CORRECT — explicit types, no any
import type { Task, TaskStatus } from '@/lib/types';

export function getStatusLabel(status: TaskStatus): string {
  const labels: Record<TaskStatus, string> = {
    TODO: 'To Do',
    IN_PROGRESS: 'In Progress',
    DONE: 'Done',
  };
  return labels[status];
}

// ❌ WRONG — implicit any, no return type
export function getStatusLabel(status) {
  return statusMap[status];
}
```

### React Components
```typescript
// ✅ CORRECT — typed props, Server Component by default
interface TaskCardProps {
  task: Task & { assignee: User | null };
}

export default function TaskCard({ task }: TaskCardProps) {
  return (
    <div className="rounded-lg border border-slate-200 bg-white p-4 shadow-sm">
      <h3 className="text-sm font-medium text-slate-900">{task.title}</h3>
    </div>
  );
}

// ❌ WRONG — untyped props, unnecessary client directive
"use client";
export default function TaskCard({ task }: any) { ... }
```

### API Routes
```typescript
// ✅ CORRECT — typed, error handled, correct status codes
export async function PATCH(
  req: Request,
  { params }: { params: { id: string } }
): Promise<Response> {
  try {
    const body: { status: TaskStatus } = await req.json();
    const task = await prisma.task.update({
      where: { id: params.id },
      data: { status: body.status },
    });
    return Response.json({ data: task }, { status: 200 });
  } catch (error) {
    return Response.json({ error: 'Failed to update task' }, { status: 500 });
  }
}
```

### Tailwind Classes
```typescript
// ✅ CORRECT — use cn() for conditional classes
import { cn } from '@/lib/utils';

const statusStyles: Record<TaskStatus, string> = {
  TODO: 'bg-slate-100 text-slate-700',
  IN_PROGRESS: 'bg-yellow-100 text-yellow-800',
  DONE: 'bg-green-100 text-green-800',
};

<span className={cn('rounded px-2 py-0.5 text-xs font-medium', statusStyles[task.status])}>
  {getStatusLabel(task.status)}
</span>

// ❌ WRONG — inline styles, no cn() for conditional logic
<span style={{ backgroundColor: status === 'DONE' ? 'green' : 'gray' }}>
```

### Prisma Usage
```typescript
// ✅ CORRECT — singleton import, typed result, includes relation
import { prisma } from '@/lib/prisma';

const tasks = await prisma.task.findMany({
  include: { assignee: true },
  orderBy: { createdAt: 'desc' },
});

// ❌ WRONG — direct instantiation
import { PrismaClient } from '@prisma/client';
const db = new PrismaClient();
```

## Dashboard Page — Special Notes

The dashboard (`src/app/dashboard/page.tsx`) is the most important page in TestApp. It must:

- Display **all tasks** with their title, status badge, and assignee name
- Be a **Server Component** — fetch data directly via Prisma, not via client-side API calls
- Group or sort tasks in a useful way (e.g., by status: Todo → In Progress → Done)
- Update reactively when the user changes a task status (use `router.refresh()` from Next.js after mutations in child Client Components)
- Remain fast and simple — no pagination needed for MVP (teams of 3–10 people generate very few tasks)
